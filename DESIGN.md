# KBO 매니저 — 1단계 설계 문서

> 목표: UI 최소화. **선수 데이터 모델**과 **경기 시뮬레이션 엔진**을 제대로 설계한다.
> 기술스택: Python 3.10+ (표준 라이브러리만, 콘솔 기반, 로직/출력 완전 분리)

## 1. 기존 코드(JS 버전)에서 살릴 것 / 버릴 것

### 살릴 것
| 자산 | 처리 방법 |
|---|---|
| 실명 로스터 (10개 구단, 약 200명) | `data/players.json`으로 변환. 새 능력치 스키마에 맞게 확장 (송구 신규 부여, 투수 구위→구속/구위 분리, 좌/우 타·투 추가) |
| KBO 규칙 구현 지식 (12회 무승부, 144경기, 규정타석/이닝, PS 사다리) | 엔진 규칙으로 재사용. 포스트시즌은 2단계 |
| "풀시즌 돌려서 리그 지표 vs 실제 KBO 비교" 검증 방식 | `scripts/calibrate.py`로 정식 밸런싱 하네스화 |
| `kbo-manager/` 웹 UI 일체 | 삭제하지 않고 보존만. 4단계(웹 UI)에서 화면 레퍼런스로 재활용 |

### 버릴 것
| 자산 | 버리는 이유 |
|---|---|
| 기존 타석 확률 모델 (K→BB→안타 단발 롤) | 얕음. 리그 평균 기준점이 없어 능력치 인플레에 취약, 수비·구장·피로·좌우 매치업 미반영 → **log5 매치업 모델**로 교체 |
| JS 게임 로직 전체 | Python으로 재작성 (엔진과 UI가 한 파일에 뒤엉켜 있어 이식 대상이 아님) |
| 수비 무시 BABIP, 고정 진루 확률 | 수비/송구 능력치가 실제로 결과에 개입하는 모델로 교체 |

## 3. 데이터 모델

### 3.1 선수 (player.py)

```python
class Hand(Enum): L, R, S              # 좌/우/스위치

class Position(Enum):
    C, FIRST, SECOND, THIRD, SS, LF, CF, RF, DH,   # 야수
    SP, RP, CL                                       # 투수 보직

@dataclass
class Contract:
    salary: float          # 연봉 (억원)
    years_left: int        # 잔여 계약기간

@dataclass
class BatterRatings:       # 0~100, 리그 평균 = 50
    contact: int           # 컨택  → 삼진 회피, BABIP
    power: int             # 파워  → 홈런율, 장타 분포
    eye: int               # 선구안 → 볼넷, 삼진 회피 보조
    speed: int             # 주루  → 도루, 추가진루, 병살 회피, 내야안타
    fielding: int          # 수비  → 팀 수비력(BABIP 억제, 병살 전환)
    arm: int               # 송구  → 주자 추가진루 억제, (포수) 도루 저지

@dataclass
class PitcherRatings:      # 0~100, 리그 평균 = 50
    velocity: int          # 구속   → 삼진, 피안타 억제
    control: int           # 제구   → 볼넷/사구 억제, 피홈런 억제 보조
    stuff: int             # 구위   → 피안타(BABIP) 억제, 피홈런 억제
    stamina: int           # 스태미나 → 투구수 한계 (피로 시작점)
    breaking: int          # 변화구 → 삼진 보조, 땅볼 유도(병살↑, 피홈런↓)

@dataclass
class Player:
    pid: str
    name: str              # 실명
    age: int
    position: Position     # 주 포지션
    bats: Hand             # 타석 방향
    throws: Hand           # 투구 방향
    contract: Contract
    batting: BatterRatings | None      # 투수도 미약한 타격치 보유 가능(당장은 None)
    pitching: PitcherRatings | None
    condition: int = 100   # 당일 컨디션 (2단계에서 변동)
    season_batting: BattingLine
    season_pitching: PitchingLine
```

### 3.2 기록 (stats.py)

```python
@dataclass
class BattingLine:   # 원시 카운트만 저장, 지표는 property로 파생
    pa, ab, h, b2, b3, hr, bb, hbp, so, gdp, sf, sb, cs, r, rbi
    # 파생: avg, obp, slg, ops

@dataclass
class PitchingLine:
    g, gs, w, l, sv, hld, outs, h, hr, bb, hbp, so, r, er, pitches
    # 파생: ip, era, whip, k9, bb9
```
경기용 박스스코어는 동일 클래스의 "1경기 인스턴스"를 쓰고, 경기 종료 시 시즌 누계에 합산한다 (경기 기록과 시즌 기록이 절대 어긋나지 않음).

### 3.3 팀 (team.py)

```python
@dataclass
class Team:

    park: ParkFactor        # 구장별 홈런/장타 보정 (잠실 넓음 ↓, 대구 ↑ 등)
```

## 4. 경기 시뮬레이션 엔진 (핵심)

### 4.1 확률 파이프라인 — log5 매치업 모델

야구 시뮬의 표준인 **odds-ratio(log5) 방식**을 쓴다. 핵심 아이디어:

1. **능력치 → 개인 진짜 실력치(true-talent rate) 변환**
   능력치 50 = 리그 평균이 되도록 각 이벤트율을 매핑한다.
   ```
   타자 삼진율  = f(컨택 70%, 선구안 30%)   # 리그 평균 18.5%, 능력치 ±50당 약 ±9%p
   타자 볼넷율  = f(선구안)                  # 리그 평균 9.0%
   타자 홈런율  = f(파워 80%, 컨택 20%)      # 리그 평균 2.2%/PA
   타자 BABIP  = f(컨택, 주루(내야안타))     # 리그 평균 .310
   투수 탈삼진율 = f(구위 40%, 구속 35%, 변화구 25%)
   투수 볼넷율  = f(제구)
   투수 피홈런율 = f(구위 50%, 변화구 30%, 제구 20%)
   투수 피BABIP = f(구위, 구속)
   땅볼성향    = f(변화구, 구위)             # GB%/FB%/LD% 분포
   ```
2. **매치업 결합 (log5)** — 타자율 B, 투수율 P, 리그 평균 L에 대해
   ```
   odds = (B/(1-B)) × (P/(1-P)) ÷ (L/(1-L))
   확률 = odds / (1 + odds)
   ```
   → 평균 투수를 만난 타자는 자기 실력대로, 강한 투수를 만나면 그만큼 눌린다.
   리그 평균이 수식에 내장되어 **능력치 인플레가 일어나도 리그 지표가 통제됨** (기존 모델의 최대 약점 해결).
3. **상황 보정 계수** (odds에 곱셈으로 적용)
   - 좌우 플래툰: 같은 손 매치업이면 타자 불리 (K↑ BB↓ HR↓, 스위치 히터는 항상 유리한 쪽)
   - 투수 피로: 투구수가 스태미나 한계 초과 시 구속/제구/구위 실효치 하락 (아래 4.3)
   - 파크팩터: 구장별 HR/2루타 배율
   - 컨디션: 당일 ±5% 수준 (2단계에서 본격화)

### 4.2 타석 해결 트리 (plate_appearance.py)

한 타석은 아래 순서로 단계 해결되어 `PAResult`를 반환한다:

```
① 사구(HBP)?          — 투수 제구 기반 (리그 평균 ~1.1%/PA)
② 볼넷? 삼진?          — log5 매치업 확률
③ 인플레이 → 홈런?     — 파워 vs 구위/변화구 매치업 + 파크팩터
④ 타구 유형 결정       — 땅볼/뜬공/라인드라이브 (투수 땅볼성향 + 타자 파워)
⑤ 안타 판정 (BABIP)    — 타자 컨택·주루 vs 투수 구위 vs ★수비팀 수비력★
   · 땅볼: 내야 수비합 / 라인드라이브: 안타 확률 최고 / 뜬공: 외야 수비
⑥ 안타면 종류 분화      — 단타/2루타/3루타 (파워·주루·타구유형·구장)
⑦ 아웃이면 부수 효과    — 병살(땅볼+1루주자+2아웃 미만, 타자 주루 vs 내야수비)
                        희생플라이(뜬공 깊이+3루주자, 외야 송구 vs 주루)
                        진루타(땅볼 진루)
⑧ 주자 처리 (baserunning.py)
   · 단타에 2루주자 홈 쇄도? → 주루 vs 외야 송구
   · 1루→3루? 도루 시도/성공? → 주루 vs 포수 송구
```

이 트리 덕분에 삼진/볼넷/사구/안타/2·3루타/홈런/땅볼/뜬공/병살/희생플라이가
전부 **개별 능력치의 결과물**로 도출된다. (요청 스펙의 이벤트 전종 커버)

### 4.3 투수 피로 모델 (pitching_manager.py)

- 타석당 투구수를 결과 기반으로 산출 (삼진 ~4.8구, 볼넷 ~5.5구, 인플레이 ~3.4구)
- 한계 투구수 = 스태미나 매핑 (스태미나 80 → 약 105구, 불펜은 25~35구)
- 한계 초과분에 비례해 구속/제구/구위 실효치 하락 → 자연스럽게 "6회 무너지는 선발" 발생
- 교체 AI: 투구수·실점·이닝·점수차·불펜 상태로 판단. 세이브 상황 마무리 투입,
  등판 간격(연투 페널티)은 시즌 모드에서 반영

### 4.4 경기 러너 (game.py)

- 9이닝, 동점 시 12회까지 연장 → KBO 규정대로 **무승부** 허용 (옵션으로 끝장승부)
- 말공격 끝내기 즉시 종료, 9회말 리드 시 생략
- 승/패/세이브/홀드 투수 판정 (리드 추적 기반 — 기존처럼 근사치가 아니라 정확한 기록)
- 반환: `GameResult(이닝별 득점, 팀 안타/실책, 선수별 박스스코어, 투수별 기록, 결승타 등)`
- **이벤트 로그**: `list[PlayEvent]`를 함께 반환 (타석 결과의 구조화 데이터).
  1단계에선 콘솔 텍스트 중계 소스, 4단계에선 웹 매치뷰 데이터가 된다.

## 5. 밸런싱 (scripts/calibrate.py)

목표: 리그 전체 N시즌 시뮬 → 아래 실제 KBO 수준(최근 수년 평균대)에 수렴.

| 지표 | 목표 범위 |
|---|---|
| 리그 타율 | .260 ~ .275 |
| 리그 출루율 / 장타율 | .335~.350 / .385~.410 |
| 리그 ERA | 4.2 ~ 4.8 |
| 경기당 득점 (팀당) | 4.6 ~ 5.2 |
| 경기당 홈런 (팀당) | 0.75 ~ 1.0 |
| K/9 · BB/9 | 7.0~8.0 · 3.3~4.0 |
| 도루 성공률 | 68 ~ 75% |
| 병살 (팀당 시즌) | 95 ~ 120 |

방법:
1. `calibrate.py`가 시드 고정으로 풀 리그 3~5시즌 시뮬 → 목표 대비 편차 표 출력
2. 편차가 큰 지표의 `TUNE` 상수만 수정 → 재실행 (상수가 한 파일에 모여 있어 반복이 빠름)
3. 분포 검증도 포함: 최고 타자(능력 90대)가 타율 .330±, 홈런 40± / 에이스 ERA 2점대 초중반이 나오는지,
   리그 1~10위 팀 승수 분포가 55~95승 사이로 퍼지는지 확인
4. `tests/test_probability.py`에 "평균 능력치 타자×평균 투수 = 리그 평균률" 수렴 테스트를 넣어
   이후 수정에서 밸런스가 깨지면 테스트가 잡아낸다

## 6. 선수 데이터 변환 계획 (data/players.json)

- 기존 JS 로스터(실명 200명)를 기반으로 스키마 확장:
  - 타자: `송구(arm)` 신규 부여, `bats`(좌/우/스위치) 부여 — 실제 선수 타석 방향 반영 (김현수 좌, 홍창기 좌, 김도영 우, 박해민 좌 등)
  - 투수: 기존 `구위(VEL)` → `구속(velocity)` + `구위(stuff)`로 분리, `변화구/제구/스태미나` 유지, `throws` 부여 (류현진 좌, 문동주 우 등)
  - 공통: `연봉`(기존 산식 유지) + `계약기간`(1~4년, 나이/등급 기반 부여) 추가
- 로스터 확충: 팀당 약 20명 → **야수 14명 + 투수 11명 = 25명** 수준으로 실존 선수 보강
  (백업 포수/내야 유틸/불펜이 부족하면 경기 엔진의 교체 로직이 비현실적이 되므로 1단계에서 확충)

## 7. 단계 로드맵 (이후 단계 미리보기)

| 단계 | 내용 |
|---|---|
| **1단계 (지금)** | 데이터 모델 + 경기 엔진 + 박스스코어 + 밸런싱 하네스 |
| 2단계 | 시즌 운영: 일정/순위/개인 타이틀/포스트시즌, 컨디션·부상·등판간격 (+에이징 커브 선행) |
| 3단계 | 구단 운영. **순서 조정**: 연봉/계약 → 드래프트 → FA → 트레이드, +2군·재정 |
| 4단계 | 웹 UI: 엔진을 그대로 두고 표현 계층만 교체 (기존 kbo-manager UI 참고) |

**3단계 순서 조정 근거** (드래프트 5→2번 앞당김): FA 보상과 트레이드가 둘 다
드래프트 지명권에 의존하고, 리그 전력 균형(동적 예산 순환)이 '돈 + 드래프트
역순지명' 두 축을 다 갖춰야 검증된다. 따라서 **드래프트 지명권을 평가 가능한
자산으로 취급**한다 — 연봉/계약 설계 시점부터 지명권이 이후 가치평가·FA 보상·
트레이드에서 자산으로 다뤄질 훅을 남긴다 (구현은 드래프트 단계, 지금은 훅만).
