# KBO 매니저 — 4단계 웹 UI 설계 (FM식 단장 모드)

> 플레이어가 10개 구단 중 하나를 맡아 운영하고, 나머지 9팀은 기존 AI가 운영.
> 경기는 시뮬, 플레이어는 결과 확인 + 자기 팀 의사결정.
>
> ※ 이 문서는 **제안**이다. 승인 후 구현. UI는 '옳은 답' 검증표가 없으므로
> 구현 후 스크린샷 → 사용자 피드백 루프로 다듬는다.

## 0. 기술 스택 / 분리 원칙

- **백엔드 FastAPI**: 기존 Python 엔진 위 **얇은 API 껍질**. 게임 로직은 절대
  UI/API에 넣지 않는다 — API는 엔진 함수를 '호출만' 하고 상태를 JSON으로 비춘다.
- **프론트 React** (Vite): 데이터 많은 관리 화면(순위표/로스터/시장) 중심.
- **기존 계층분리 확장**: `engine`(순수) / `league`(러너) / `io`(콘솔 표현) 구조에서
  API는 `io`와 병렬의 **또 하나의 표현 계층**이다. 콘솔 스크립트는 그대로 산다.

### 엔진은 안 건드린다 — 단, `league` 러너에 최소 훅 2개 (정직하게 명시)

게임 로직 수정이 아니라 **호출 단위/제어권 훅**이며, 콘솔 파이프라인과 검증
스크립트는 동작 불변이다:

1. **일 단위 스텝**: `SeasonRunner.run()`은 시즌을 통째로 돌린다. UI의 진행
   컨트롤(하루/시리즈/구간)을 위해 `run()`을 내부 분해한 `step_day()`를 추가
   (run은 step_day 반복으로 재구현 — 기존 호출자 무영향).
2. **유저 라인업 존중**: 시즌 러너가 매일 `build_default_lineup()`으로 라인업을
   재구성한다. 유저 팀은 유저가 짠 타순/로테이션을 쓰도록 `Team.user_managed`
   플래그 + "유저 라인업 유지(부상자만 자동 대체)" 분기.

오프시즌 개입은 훅이 **필요 없다** — 체인이 이미 함수 단위(`run_trades` /
`run_fa_market` / `run_draft` / `offseason_finance_tick`)로 쪼개져 있어,
API 레이어가 단계 사이에 멈추고 사람 입력을 끼워 넣는 오케스트레이션만 한다.

## 1. 게임 세션 / 상태

- **단일 세이브 인메모리 세션** (MVP): 서버가 리그 상태(팀/선수/시즌 진행 위치/
  오프시즌 단계)를 들고 있고, 파일 저장/불러오기(pickle) 제공. 멀티 세이브는 후순위.
- 게임 루프 상태머신:
  ```
  SEASON(day i/144) → SEASON_END(순위 확정) → POSTSEASON
    → OFFSEASON: aging → trade(유저 개입) → fa(유저 개입) → draft(유저 개입)
    → finance → 새 시즌
  ```
  유저 개입 단계에서만 멈추고, 나머지는 자동 통과 + 요약 리포트.

## 2. 화면 구성 (텍스트 목업)

### 2.1 대시보드/홈 — 게임의 기본 착지점 [MVP-1]
```
┌─ KIA 타이거즈 (유저) ── 2026시즌 87일차 ─────────────────────┐
│ 순위 3위 (52승 2무 33패, 1위와 4.5경기차)     [팀 로고/색]    │
│ 최근 5경기: ○○●○●   다음 상대: LG (홈, 3연전)              │
├──────────────┬──────────────────────────────────────────────┤
│ 진행 ▶       │ 오늘의 알림                                   │
│ [하루]       │ · 김도영 3안타 (타율 .352, 리그 2위)          │
│ [다음 시리즈]│ · 투수 이의리 부상 복귀 (12일 결장)           │
│ [한 달]      │ · 리그: 안우진 시즌 10승                      │
│ [시즌 끝까지]│                                               │
└──────────────┴──────────────────────────────────────────────┘
```
보여주는 것: 내 팀 성적·순위, 다음 경기, 알림(부상/기록). 하는 것: 시즌 진행.

### 2.5 일정/결과 [MVP-1]
```
9월 12일  KIA 5 : 3 LG   [박스스코어 보기]
박스스코어 = 기존 boxscore_text의 데이터 버전 (이닝별 R/H/E, 타자/투수 라인,
승/패/세, 결승타). [MVP-3] 타석별 이벤트 로그 중계 뷰.
```

### 2.6 오프시즌 허브 [MVP-2] — 사람 개입의 본체
```
오프시즌 2026 → [1 에이징 리포트] → [2 트레이드] → [3 FA] → [4 드래프트] → [5 재정]
                    (자동+요약)      (개입)        (개입)      (개입)       (자동+요약)
```

**(a) 에이징 리포트** (자동, 확인만): 내 팀 성장/쇠퇴 Top5, 은퇴자 명단.

**(b) 트레이드 — 수신함 + 발신 제안 둘 다**:
```
[받은 제안]  SSG가 제안: 최정(40, 3B) ↔ 우리 유망주 김택연 + 2R픽   [수락/거절]
[제안하기]  상대팀 선택 → 상대 로스터에서 원하는 선수 → 내가 줄 자산 선택
            → AI가 자기 GMView(시간선호+노이즈)로 3조건 평가 → 수락/거절/
            "지명권을 얹으면 고려" 카운터
내 팀 니즈 힌트: 포지션별 뎁스 바 (C 약함 / SS 보통 / SP 강함)
```

**(c) FA — 봉인식 경매에 사람 참여**:
```
FA 시장 명단: 이름 나이 등급 포지션 OVR 적정AAV 보상금 [입찰]
내 입찰: 선수 선택 → AAV 입력(적정가 가이드 표시) → 제출
(우리 팀 잔류 대상자에겐 잔류 오퍼 여부/금액도 결정)
[시장 해결] 버튼 → AI 입찰과 함께 봉인식 해결 → 결과 리포트
  ※ 선수 선택은 기존 appeal 모델 그대로 — 돈만 많이 쓴다고 못 데려옴
```

**(d) 드래프트 — 내 순번 직접 지명**:
```
드래프트 보드: 순번 진행 로그 + 남은 유망주 표
  이름  나이 포지션  스카우팅 평가(노이즈 낀 관측치!)  아키타입 힌트
내 차례가 되면 [지명] 활성화. 진짜 재능은 숨김 — 스카우팅 평가만 보고 도박.
니즈 힌트: 내 로스터 구멍 포지션 강조.
```

### 2.7 시즌 리더보드/기록실 [MVP-3]
타율/홈런/타점/도루/ERA/다승/세이브/탈삼진 리더 (기존 leaders_text 데이터 버전),
포스트시즌 브래킷 뷰, 역대 우승 연혁.

## 3. API 초안 (얇은 껍질)

```
POST /game/new {tid, seed}         GET  /game/state        (상태머신 위치·요약)
POST /game/save · /game/load
POST /sim/advance {unit: day|series|month|season_end}
GET  /standings                    GET  /schedule?from&to
GET  /teams/{tid}/roster           GET  /players/{pid}
PUT  /my/lineup {order[], rotation[], closer}          [MVP-2]
GET  /offseason/state              POST /offseason/advance (다음 단계로)
GET  /trade/proposals              POST /trade/respond {id, accept}
POST /trade/propose {give[], want[]}                    [MVP-2]
GET  /fa/pool                      POST /fa/bid {pid, aav}  POST /fa/resolve
GET  /draft/state                  POST /draft/pick {pid}
```
응답은 전부 엔진 데이터의 JSON 직렬화 (Player/Team → dict 변환기 하나).

## 4. 사람 개입 지점 — 기존 로직에 끼우는 방법

| 지점 | 방식 |
|---|---|
| 라인업/로테이션 | `Team.user_managed` — 러너가 유저 편성 유지(부상자만 자동 대체) |
| 트레이드 (수신) | `run_trades`의 매칭 루프에서 **유저 팀이 낀 쌍은 자동 실행 대신 제안 큐에 적재** → 유저 수락 시 실행. AI끼리는 기존대로 |
| 트레이드 (발신) | 유저가 자산 조합 지정 → 상대 AI가 자기 GMView로 기존 3조건 평가 (같은 함수 재사용) → 수락/거절/카운터(지명권 요구) |
| FA | 유저 팀은 AI 자동입찰에서 제외, 유저가 입찰서 제출 → 기존 봉인식 해결에 합류. 선수의 선택은 기존 appeal 그대로 |
| 드래프트 | `run_draft`를 픽 단위 제너레이터로 진행 — AI 픽은 자동, **유저 순번에서 멈춰** 선택 대기. 유저에겐 스카우팅 관측치만 노출(진짜 재능 숨김 유지) |

## 5. MVP 단계 (사용자 제안에 동의 + 세분화)

| 단계 | 내용 | 완료 기준 |
|---|---|---|
| **MVP-1 관전하는 단장** | 팀 선택 → 시즌 진행(하루/시리즈/월/끝까지) → 대시보드·순위표·일정/결과(박스스코어)·로스터·선수 상세 (전부 읽기 전용). 오프시즌은 전자동 통과 + 단계별 요약 리포트. 저장/이어하기 | "한 팀을 골라 여러 시즌을 돌리며 구경할 수 있다" |
| **MVP-2 개입하는 단장** | 라인업/로테이션 편집 + 오프시즌 개입 4종(트레이드 수신/발신, FA 입찰, 드래프트 지명) + 개입 지점 일시정지 플로우 | "내 결정이 팀 성적을 바꾼다" |
| **MVP-3 질감** | 타석 이벤트 중계 뷰, 리더보드/기록실, 나이곡선·재정 리포트, 뉴스피드, 멀티 세이브 | "게임답다" |

- MVP-1을 먼저 완결하는 이유: 개입 UI(2차)는 화면·상태머신 위에서만 만들 수
  있고, 관전 루프가 돌아야 스크린샷 피드백 루프를 조기에 시작할 수 있다.
- 포스트시즌은 MVP-1에서 자동 시뮬 + 결과 요약(브래킷 뷰는 MVP-3).

## 6. 폴더 구조

```
web/
├─ backend/
│  ├─ main.py          # FastAPI 앱 (라우터 조립)
│  ├─ session.py       # 게임 세션·상태머신·저장/로드
│  ├─ serializers.py   # Player/Team/GameResult → JSON dict
│  └─ routers/         # game / sim / team / offseason
└─ frontend/           # React (Vite). pages/ = §2 화면 1:1
```

### 승인받을 핵심 결정
1. **FastAPI 얇은 껍질 + React**, API는 `io`와 병렬인 표현 계층 (게임 로직 없음)
2. `league` 러너에 훅 2개만: **일 단위 스텝** + **유저 라인업 존중** (엔진 불변,
   콘솔/검증 스크립트 동작 불변)
3. 오프시즌 개입 = 기존 함수 사이 오케스트레이션 (트레이드 제안 큐 / FA 입찰
   합류 / 드래프트 픽 단위 정지) — 로직 재사용, 유저에겐 스카우팅 관측치만 노출
4. **MVP-1 = 관전하는 단장** (읽기 전용 + 시즌 진행 + 저장), 개입은 MVP-2
5. 단일 세이브 인메모리 세션 + 파일 저장 (멀티 세이브 후순위)
