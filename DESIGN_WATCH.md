# KBO 매니저 — 경기 관전 설계 (웹 UI MVP-2: 2D 그라운드 뷰)

> 경기가 '숫자로만' 존재하는 문제를 푼다. 엔진은 매 타석을 8단계 트리로,
> 투수 피로·주자·병살·실책까지 계산한다 — 그 드라마를 **이벤트 스트림**으로
> 노출하고 OOTP식 2D 그라운드 뷰로 재생한다. 라인업 편집·오프시즌 개입은
> MVP-3으로 이월.
>
> ※ 이 문서는 **제안**이다. 승인 후 구현.

## 0. 원칙

- ★**엔진 로직 불변**: '이미 계산하는 값을 기록해 내보내는' 것만. 기록은
  rng를 소비하지 않는다 → 시드 결과 불변 (검증 §4).
- ★**표현 계층 독립**: 스트림에는 '무슨 일이 일어났는지'(사실)만. '어떻게
  그릴지'는 클라이언트 몫 — 나중에 Unity 3D 클라이언트가 같은 스트림을
  소비한다.
- 관전 = **방금 시뮬된 경기의 리플레이**. 진행(advance)이 경기를 결정하고,
  관전은 기록된 스트림을 재생한다 → "건너뛰기 해도 결과 동일"이 구조적으로
  보장된다 (결과는 이미 확정).
- 관전 범위: **내 팀 경기만** 스트림 기록 (메모리 절약 + 요구사항).

## 1. 이벤트 스트림 스키마 (사실만)

경기당 `events: list[dict]`, 시간 순서. 타입 5종:

```jsonc
// (1) 타석 — 스트림의 본체
{"t": "pa", "inning": 3, "half": "top",            // 초/말
 "outs": 1, "score": [2, 1],                        // 타석 시작 시점
 "batter": {"pid", "name", "order": 4},
 "pitcher": {"pid", "name", "pitches": 68, "fatigued": true},
 "outcome": "2B",          // K/BB/HBP/1B/2B/3B/HR/GO/FO/LO/DP/SF/E
 "ball_type": "LD",        // GB/LD/FB ("" = 인플레이 아님)
 "pitches": 5,             // 이 타석 투구 수 (엔진 계산값)
 "bases_before": [pid|null, pid|null, pid|null],   // [1루,2루,3루]
 "bases_after":  [ ... ],
 "scored": [{"pid", "name"}],                      // 홈인 순서
 "outs_added": 1, "rbi": 1,
 "seed": 173042}           // 연출용 결정론 시드 (모든 클라이언트가 동일 연출)

// (2) 도루 시도
{"t": "steal", "inning", "half", "outs", "runner": {"pid","name"},
 "from": 1, "to": 2, "success": true}

// (3) 투수 교체
{"t": "pitch_change", "inning", "half", "team": "KIA",
 "in": {"pid","name"}, "out": {"pid","name"}}

// (4) 이닝 전환   (5) 경기 종료
{"t": "half_end", "inning", "half", "score": [..]}
{"t": "game_end", "score": [..], "innings": 10, "tie": false,
 "decisions": {"W","L","SV"}}
```

- **주자 이동**은 `bases_before/after + scored`의 사실 조합으로 완전 결정된다
  (누가 어느 베이스에서 어디로 — 클라이언트가 diff로 도출). 이동 경로를
  스트림에 넣지 않는 이유: 그건 '어떻게 그릴지'다.
- **타구 방향**: 엔진이 계산하지 않는 값 → 스트림에 없음. 클라이언트가
  `seed`로 결정론 연출 (모든 클라이언트 동일 화면, 결과 무영향).
- **부상**: 경기 중이 아니라 경기일 종료 후 일일 판정(기존 설계)이라 경기
  스트림에 없다 — 대시보드 뉴스가 담당 (현행 유지).
- 각 이벤트에 `text`(한국어 중계 한 줄)를 **직렬화 계층**이 첨부 — 사실이
  아니라 표현 편의이며 Unity는 무시 가능함을 명시.

## 2. 볼카운트 연출 — rng 안전이 생명

엔진은 투구별 시뮬을 하지 않는다 (타석당 투구 수만 계산). 볼카운트 진행은
**연출용 재구성**이다:

- 결과와 투구 수에 **정합하는** 자연스러운 B/S 시퀀스를 생성
  (삼진이면 마지막이 S, 볼넷이면 B 4개 포함, 인플레이면 마지막이 X 등).
- ★생성 위치는 **엔진 밖** (백엔드 직렬화 계층). 엔진 rng를 쓰면 난수
  스트림이 이동해 시드 결과가 깨진다 → 이벤트별 `seed`로 만든 **독립
  Random**을 사용. 결과 무영향 + 재현 가능 + 전 클라이언트 동일.

## 3. 기록 지점 (엔진 내 계측 — 로직 불변)

`GameSimulator`에 `record_struct` 플래그 추가. 기존 텍스트 이벤트(`_ev`)가
기록되는 **같은 지점**에서 구조화 이벤트를 append (스냅샷 읽기만, 분기·rng
소비 없음):

| 지점 | 이벤트 |
|---|---|
| `_half` 타석 직전/직후 | bases 스냅샷 → `pa` (PAResult가 outcome·scored·pitches 다 가짐) |
| `_half` 도루 분기 | `steal` |
| `maybe_change` 반환 시 | `pitch_change` |
| 하프이닝 종료 / `_finish` | `half_end` / `game_end` |

- `SeasonRunner.step_day`가 유저 팀 경기에만 `record_struct=True` 전달
  (`user_managed` 플래그 재사용 — 신규 훅 아님).
- 세션이 `watch_events[(day, game_idx)]`로 보관, 시즌 교체 시 정리.
- API: `GET /api/watch/{day}/{game_idx}` → `{meta, events[]}`.
  유저 팀 경기가 아니면 404 ("관전 불가 — 결과만 제공").

## 4. 검증 (요구사항 그대로)

1. **정합성**: 이벤트 폴드(득점 합·아웃 합·타자별 안타/홈런/타점 집계)가
   같은 경기 박스스코어와 **정확히 일치** (자동 테스트).
2. **재현성**: 같은 시드 → 같은 스트림 (기록이 rng 무소비 → 자동 보장, 테스트).
3. **건너뛰기 동일**: 관전=리플레이 구조라 자명 — 기록 ON/OFF로 경기 결과가
   안 변하는 것을 테스트로 가드.
4. **엔진 회귀 없음**: 55개 테스트 + seed 2026 = KIA 97승 재확인.

## 5. 프론트 — 2D 그라운드 뷰 (SVG)

```
┌────────────────────────────────────┬──────────────┐
│  [스코어버그] 5회말 · 2아웃 · B2-S1 │  중계 로그    │
│   KIA 3 : 4 SSG   ◆◇◆(주자 다이어그램)│  5회말        │
│ ┌────────────────────────────────┐ │  김도영 좌중간 │
│ │        2D 필드 (SVG)           │ │  2루타, 1루    │
│ │   외야 잔디 + 내야 다이아몬드   │ │  주자 홈인     │
│ │   수비 9명(포지션 점), 담장 호   │ │  (KIA 3-4)    │
│ │   주자 토큰(베이스 이동 애니)    │ │  ...           │
│ │   타구 애니: GB 낮은 직선 /      │ │               │
│ │   FB 포물선 / LD 빠른 직선 /     │ │               │
│ │   HR 담장 밖 + 이펙트            │ │               │
│ └────────────────────────────────┘ │               │
│  [타자 카드]        [투수 카드]     │               │
│  김도영 .335 3타수2안타│ 폰세 82구 피로▼│              │
├────────────────────────────────────┴──────────────┤
│  ▶재생/⏸ · 속도 1x/2x/4x · 다음 타석 ⏭ · 다음 이닝 ⏭⏭ · 결과로 건너뛰기 │
└───────────────────────────────────────────────────┘
```

- **필드**: SVG 벡터 (깔끔한 2D — 어설픈 3D 배제). 수비 9명 고정 배치,
  베이스 4개, 파울라인, 담장 호.
- **재생기**: 이벤트 배열을 타이머로 소비. 타석 = 볼카운트 진행(투구당
  ~0.5초/1x) → 타구 애니(~1.2초) → 주자 이동 → 상태 갱신. 속도는 타이머
  배율. '다음 타석/이닝/결과로'는 인덱스 점프 + 상태 즉시 반영.
- **타자/투수 카드**: 능력치 요약 + **오늘 성적** (스트림 폴드로 실시간 누적).
- **진입**: 일정·결과에서 내 팀 경기 클릭 → [관전] / [결과만] 선택.
  대시보드 진행 버튼은 현행 유지 (진행 후 관전은 결과 화면에서).

### 승인받을 핵심 결정
1. 스트림 = **사실만** (bases before/after·scored·outcome·타입 5종) + 이벤트별
   `seed`로 전 클라이언트 동일 연출 — 타구 방향·이동 경로는 클라이언트 도출
2. 볼카운트는 **엔진 밖에서 독립 rng로 재구성** (결과 정합·rng 안전)
3. 관전 = **방금 시뮬된 경기 리플레이** (내 팀 경기만 기록) — 건너뛰기 동일성
   구조 보장
4. 필드는 **SVG 벡터 2D**, 재생 컨트롤 4종 (재생/속도/타석/이닝/결과 점프)
5. `text` 중계 문구는 직렬화 계층의 표현 편의 (Unity 무시 가능)
