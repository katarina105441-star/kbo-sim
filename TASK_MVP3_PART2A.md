# MVP-3 Part 2A — 실시간 경기 진행 + 수동 투수 교체

기준 브랜치: `feat/mvp3-live-management`
기준 커밋: `f57abab8d8dca5d6bca55ac9846c58c7803dd981`

## 목적

현재 관전 모드는 이미 끝난 경기를 재생한다. 이번 단계에서는 사용자의 당일 경기를 실제로 한 타석씩 진행하고, 다음 타석 시작 전에 수동 투수 교체를 지시할 수 있게 한다.

이번 단계는 향후 대타·대주자·대수비를 추가할 수 있는 상태머신 기반을 만드는 작업이다.

## 반드시 지킬 원칙

- 기존 `GameSimulator.run()` 호출 결과와 시드 재현성은 그대로 유지한다.
- 기존 콘솔·시즌·포스트시즌 자동 진행 경로는 변경 전과 동일하게 작동해야 한다.
- 기존 공유 RNG 모드 및 웹 격리 RNG 모드 결과를 깨지 않는다.
- Python generator를 장기 세션 상태로 저장하지 않는다. 명시적인 직렬화 가능한 상태 객체를 사용한다.
- 게임 확률과 밸런싱 상수는 변경하지 않는다.
- 한 경기 결과를 두 번 통계에 반영하지 않는다.
- 실시간 경기 완료 전에는 해당 날짜의 부상·폼 처리 및 `season.day` 증가가 일어나면 안 된다.

## 1. GameSimulator 상태머신

`kbo/engine/game.py`를 명시적 상태 기반으로 리팩터한다.

필수 공개 인터페이스 예시:

```python
sim.start()
state = sim.state()
event = sim.step_pa()
sim.force_pitcher_change("home", pitcher_pid)
result = sim.finish_auto()
```

이름은 조정 가능하지만 다음 동작은 필수다.

- `run()`은 새 상태머신을 끝까지 자동 실행하는 호환 래퍼가 된다.
- `step_pa()`는 최대 한 타석만 진행한다.
- 도루·자동 투수 교체·이닝 종료·끝내기·경기 종료를 기존 로직과 동일하게 처리한다.
- 다음 타석 전에만 수동 투수 교체 가능하다.
- 수동 교체 시 기존 `PitchingStaff.bring()`을 사용하고 구조화 `pitch_change` 이벤트를 남긴다.
- 이미 등판한 투수, 부상자, 당일 출전 불가 투수, 상대 팀 투수, 야수는 선택할 수 없다.
- 한 번 교체되어 내려간 투수는 재등판할 수 없다.
- 수동 교체 직후 같은 타석 시작 전 AI가 다시 자동 교체하지 않도록 해당 결정 지점을 한 번 소비한다.
- 상태에는 이닝, 초/말, 아웃, 주자, 스코어, 다음 타자, 현재 투수, 투구수, 사용 가능한 불펜, 종료 여부가 포함된다.

## 2. SeasonRunner 날짜 분할

`kbo/league/season.py`의 하루 진행을 다음 두 단계로 분리한다.

- 일반 경기 및 비사용자 경기 진행
- 사용자 실시간 경기 완료 후 날짜 정산

권장 구조:

```python
ctx = season.begin_day(managed_tid=user_tid)
# 비사용자 4경기는 즉시 완료
# 사용자 경기 GameSimulator는 pending으로 반환
season.complete_managed_game(ctx, result)
```

요구사항:

- 기존 `step_day()`는 내부적으로 begin/complete를 자동 호출해 결과가 기존과 같아야 한다.
- 격리 모드의 경기별 RNG 시드는 자동 진행과 실시간 진행에서 동일해야 한다.
- 사용자 경기의 수동 교체로 난수 소비가 바뀌어도 다른 4경기, 다른 팀 부상·폼은 변하지 않는다.
- 날짜 정산은 정확히 한 번만 실행한다.
- 중복 begin, 완료 전 다음 날짜 진행, 중복 complete를 명확한 오류로 차단한다.

## 3. GameSession 실시간 경기 상태

`web/backend/session.py`에 직렬화 가능한 pending live 상태를 추가한다.

필수 동작:

- 현재 날짜의 사용자 경기가 있을 때 실시간 시작 가능
- 실시간 진행 중 일반 `advance` 차단
- 실시간 경기 종료 시 결과를 `results_by_day`에 다른 경기들과 함께 저장
- 기존 스포일러 숨김 정책 유지
- 실시간 경기는 이미 사용자가 직접 본 경기이므로 종료 즉시 공개 처리 가능
- 저장 기능은 pending 상태까지 안전하게 저장·로드하거나, 이번 단계에서 불가능하다면 저장 API가 명확한 409 오류를 반환해야 한다. 조용히 깨진 pickle을 만들면 안 된다.

## 4. API

최소 API:

- `POST /api/live/start`
- `GET /api/live/state`
- `POST /api/live/step`
- `POST /api/live/pitcher`
- `POST /api/live/auto`

`POST /api/live/pitcher` 요청 예시:

```json
{"pid": "KIA_P012"}
```

오류 정책:

- 실시간 경기 없음: 404
- 잘못된 진행 상태: 409
- 잘못된 선수·중복 등판·부상·출전 불가: 422

## 5. 프론트엔드

기존 `Watch.jsx`는 완료 경기 재생용으로 유지한다.

새 화면 `LiveGame.jsx`를 추가한다.

최소 기능:

- 현재 이닝·아웃·주자·스코어 표시
- 현재 타자·투수·투구수 표시
- `다음 타석` 버튼
- `자동 진행` 버튼
- 사용자 팀이 수비 중이고 다음 타석 전일 때 `투수 교체` 버튼
- 사용 가능한 불펜 목록과 피로/OVR 표시
- 교체 성공·실패 메시지
- 경기 종료 후 기존 결과/박스스코어 화면으로 이동

이번 단계에서는 투구 단위 애니메이션을 새 실시간 API에 연결하지 않아도 된다. 한 타석 단위 진행을 우선한다.

## 6. 테스트

기존 80개 테스트 전부 통과.

신규 필수 테스트:

1. `run()`과 `step_pa()` 끝까지 실행한 결과가 같은 시드에서 동일
2. 기존 공유 모드 시드 앵커 유지
3. 웹 격리 모드에서 실시간 사용자 경기 진행 여부가 다른 4경기 결과에 영향 없음
4. 실시간 경기 완료 전 `season.day`가 증가하지 않음
5. 완료 후 날짜 정산이 정확히 한 번 실행
6. 수동 투수 교체 후 다음 타석부터 새 투수가 기록됨
7. 이미 등판한 투수 재등판 거부
8. 부상·출전 불가·야수·상대 팀 선수 거부
9. 실시간 진행 중 일반 advance 거부
10. API start/state/step/pitcher/auto 정상·오류 경로
11. 저장 정책 테스트: pending 저장 지원 또는 409 차단
12. 프론트 `npm run build` 성공
